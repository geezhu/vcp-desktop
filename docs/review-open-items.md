## 3. 评审议题

### 3.1 Gitee 镜像发布策略（可选）

**当前状态**:

1. checklist 仍有两项未勾选：同步 tag/资产到 Gitee、校验镜像一致性。
2. 项目当前决策是“Gitee 暂时冻结，不作为阻塞项”。

**候选方案**:

1. 方案 A（维持冻结）  
说明：继续仅使用 GitHub 作为唯一发布源。  
优点：流程简单、维护成本最低。  
风险：部分网络环境下载可用性差。

2. 方案 B（手动镜像）  
说明：每个版本由人工把 GitHub release 资产同步到 Gitee。  
优点：不改 CI，立刻有国内镜像。  
风险：人工操作易错、版本一致性依赖流程纪律。

3. 方案 C（自动镜像）  
说明：补充镜像同步流水线，并加入一致性校验。  
优点：长期稳定、可审计。  
风险：实现与维护成本最高。

**建议（默认）**:

1. 当前阶段选择方案 A（维持冻结），把资源集中在安装器可运行性和兼容性。
2. 设定明确解冻门槛：出现连续下载失败反馈、或进入大规模外部测试阶段，再切换 B/C。

**需要确认**:

1. 是否本阶段解冻 Gitee（是/否）。
2. 若解冻，选择 B 还是 C。

### 3.2 Linux 发行版承诺范围

**当前状态**:

1. 口径是 Linux `x86_64` + `glibc` 基线。
2. Ubuntu/Debian 目前是回归基线，不代表唯一支持发行版。

**候选方案**:

1. 方案 A（最小承诺）  
声明“首发只保证 Ubuntu 22.04+/Debian 12+ 回归通过”。  

2. 方案 B（分层承诺）  
Tier-1: Ubuntu/Debian；Tier-2: RHEL/Fedora（best effort）；其他发行版不承诺。  

3. 方案 C（广泛承诺）  
直接声明更多发行版正式支持。

**建议（默认）**:

1. 选方案 B（分层承诺），既保守又有扩展空间。

**需要确认**:

1. 是否采用 Tier 分层声明。
2. Tier-2 是否纳入 RHEL/Fedora。

### 3.4 `config.env` 修改策略

**当前状态**:

1. 已支持“生成配置”和“覆盖配置”开关（`--overwrite-config`）。
2. 仍需明确默认策略与用户预期。

**候选方案**:

1. 方案 A（默认不改已有配置）  
已有配置存在时仅提示，不覆盖。

2. 方案 B（默认覆盖）  
每次按安装器模板重写。

3. 方案 C（三态）

**建议（默认）**:

1. 选方案 A（最安全），并保留显式覆盖开关。
2. 后续需要更细粒度时演进为 C。

**需要确认**:

1. 是否保留“默认不覆盖”作为长期策略。
