## 3. 评审议题

### 3.1 Gitee 镜像发布策略（可选）

**当前状态**:

1. checklist 仍有两项未勾选：同步 tag/资产到 Gitee、校验镜像一致性。
2. 项目当前决策是“Gitee 暂时冻结，不作为阻塞项”。

**候选方案**:

1. 方案 A（维持冻结）  
说明：继续仅使用 GitHub 作为唯一发布源。  
优点：流程简单、维护成本最低。  
风险：部分网络环境下载可用性差。

2. 方案 B（手动镜像）  
说明：每个版本由人工把 GitHub release 资产同步到 Gitee。  
优点：不改 CI，立刻有国内镜像。  
风险：人工操作易错、版本一致性依赖流程纪律。

3. 方案 C（自动镜像）  
说明：补充镜像同步流水线，并加入一致性校验。  
优点：长期稳定、可审计。  
风险：实现与维护成本最高。

**建议（默认）**:

1. 当前阶段选择方案 A（维持冻结），把资源集中在安装器可运行性和兼容性。
2. 设定明确解冻门槛：出现连续下载失败反馈、或进入大规模外部测试阶段，再切换 B/C。

**需要确认**:

1. 是否本阶段解冻 Gitee（是/否）。
2. 若解冻，选择 B 还是 C。

**本轮评审结论（2026-02-25）**:

1. 结论：继续冻结（方案 A）。
2. 执行口径：Gitee 镜像不作为当前里程碑阻塞项；后续仅在触发“解冻门槛”时重开此议题。

### 3.2 Linux 发行版承诺范围

**当前状态**:

1. 口径是 Linux `x86_64` + `glibc` 基线。
2. Ubuntu/Debian 目前是回归基线，不代表唯一支持发行版。

**候选方案**:

1. 方案 A（最小承诺）  
声明“首发只保证 Ubuntu 22.04+/Debian 12+ 回归通过”。  

2. 方案 B（分层承诺）  
Tier-1: Ubuntu/Debian；Tier-2: RHEL/Fedora（best effort）；其他发行版不承诺。  

3. 方案 C（广泛承诺）  
直接声明更多发行版正式支持。

**建议（默认）**:

1. 选方案 B（分层承诺），既保守又有扩展空间。

**需要确认**:

1. 是否采用 Tier 分层声明。
2. Tier-2 是否纳入 RHEL/Fedora。

**本轮评审结论（2026-02-25）**:

1. 结论：采用方案 A（最小承诺）。
2. 对外口径：首发兼容承诺限定为 Ubuntu 22.04+ / Debian 12+ 回归通过。
3. 其他发行版：可尝试运行，但不纳入首发承诺范围。

**当前状态**:

1. 现有路径是在线初始化下载（portable runtime）。
2. 已具备缓存、resume、弱网回归脚本，但无正式“离线包产物”。

**候选方案**:

1. 方案 A（暂不提供离线包）  
保留在线路径，继续强化失败恢复与提示。

2. 方案 B（提供最小离线包）  
仅包含 runtime + 核心组件依赖。

3. 方案 C（完整离线包）  
包含插件依赖矩阵与更多可选组件。

**建议（默认）**:

1. 先 A，再在里程碑后切 B。
2. C 仅在明确内网部署需求后立项。

**需要确认**:

**本轮评审结论（2026-02-25）**:

2. 交互要求：

   - 安装器提供离线包下载跳转按钮（打开发布页/下载页）。
3. 责任边界：离线包文件由用户自行下载，不由安装器自动预置。

### 3.4 `config.env` 修改策略

**当前状态**:

1. 已支持“生成配置”和“覆盖配置”开关（`--overwrite-config`）。
2. 仍需明确默认策略与用户预期。

**候选方案**:

1. 方案 A（默认不改已有配置）  
已有配置存在时仅提示，不覆盖。

2. 方案 B（默认覆盖）  
每次按安装器模板重写。

3. 方案 C（三态）  
`keep` / `merge` / `overwrite` 三种明确策略。

**建议（默认）**:

1. 选方案 A（最安全），并保留显式覆盖开关。
2. 后续需要更细粒度时演进为 C。

**需要确认**:

1. 是否保留“默认不覆盖”作为长期策略。

**补充说明（你问的“什么时候修改”）**:

1. 触发时机：执行 `install` / `resume` 流程且进入 `S50_TOOLBOX` 配置步骤时。
2. 当前行为：
   - 若 `config.env` 不存在：生成模板并写入最小可运行字段。
   - 若 `config.env` 已存在：会按最小运行基线对关键字段执行补丁写入。
   - `--overwrite-config` 用于“强制按模板重建文件”。
3. 评审解释：此项关注的是“安装器对已有配置的改写边界”。

### 3.5 安装后自动启动策略

**当前状态**:

1. 安装器支持安装后启动（参数控制），也支持手动 `start`。
2. 仍需确定“默认行为”。

**候选方案**:

1. 方案 A（默认不自动启动）  
安装完成后提示用户手动启动。

2. 方案 B（默认自动启动）  
安装完成即拉起后端与 chat。

3. 方案 C（首次安装询问一次并记忆）  
首装弹一次选择，后续按用户偏好执行。

**建议（默认）**:

1. 选方案 C，兼顾新手体验与可控性。
2. headless 模式下默认不自动启动，除非显式参数指定。

**需要确认**:

1. 默认行为采用 A/B/C 哪个。

**本轮评审结论（2026-02-25）**:

1. 结论：安装完成后提供“运行选择框”让用户决定是否立即启动。
2. 交互建议：
   - 选项 1：立即启动后端与 VCPChat。
   - 选项 2：稍后手动启动（展示 `start` 命令/按钮）。
3. headless 约束：在无交互环境下默认不自动启动，需显式参数触发。

### 3.6 GUI 必需步骤判定规则

**当前状态**:

1. 已有 headless 阻塞提示与 resume 机制。
2. 仍需统一“什么算 GUI 必需步骤”的判定源。

**候选方案**:

1. 方案 A（固定白名单）  
由安装器内置规则定义。

2. 方案 B（能力探测）  
根据环境和任务动态判断。

3. 方案 C（插件声明 + 核心兜底）  
插件 manifest 声明 + 安装器规则兜底。

**建议（默认）**:

1. 选方案 C（长期可扩展）。
2. 当前先以 A 作为过渡，逐步迁移到 C。

**需要确认**:

1. 目标架构是否采用 C。
2. 过渡期是否接受 A -> C 的迭代路线。

**补充说明（你问的“具体是什么意思”）**:

1. 本议题的核心是：哪些步骤在 CLI/headless 下必须中断并要求用户去 GUI 完成。
2. 典型“GUI 必需步骤”示例：桌面授权弹窗确认、仅 GUI 可操作的首次授权流程、图形化组件初始化。
3. 判定规则的差别：
   - 白名单：固定规则，简单但扩展性差。
   - 能力探测：动态判断，复杂度高。
   - 插件声明+兜底：可扩展，需定义 schema 与校验。
4. 本轮状态：已澄清含义；具体采用 A/B/C 的最终架构仍可在下一轮确认。
